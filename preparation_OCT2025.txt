✅ 🔹 Spark Pipeline Pre-Deployment Checklist (PySpark + Databricks)
🧩 1. Code Quality & Standards
Check	Description
Code Review Done	Peer-reviewed for logic, coding standards, error handling, and Spark best practices.
Modular Code	Logic is organized into reusable functions/modules instead of monolithic notebooks.
No Hardcoded Values	All paths, dates, environment names, credentials are parameterized (use widgets, configs, or environment variables).
Config-driven Design	Externalize configurations (like input/output paths, table names, thresholds) into YAML/JSON/parameter tables.
Proper Comments & Logging	Key steps are logged using log4j or Python logging. Avoid print statements in production.
Version Control	Code pushed and tagged properly in Git (branch → PR → merge → tag).
⚙️ 2. Environment & Configuration Readiness
Check	Description
Cluster Configuration Validated	Check driver/executor memory, cores, and autoscaling settings appropriate for data volume.
Environment-specific Configs	Validate prod paths (e.g., ADLS/S3/HDFS) and connection strings point to production datasets.
Secrets Management	All credentials fetched from Databricks Secrets / Key Vault — no credentials in code.
Dependencies Installed	All required Python packages are installed in the cluster or included in a .whl or requirements.txt.
Job Parameters	Verify pipeline parameters are dynamic (e.g., date, batch_id, ingestion_type).
🧮 3. Data Validation Checks
Check	Description
Schema Validation	Verify schema consistency between source and target tables.
Null/Bad Data Handling	Ensure logic gracefully handles missing or malformed records.
Duplicate Handling	Validate deduplication logic is working as expected.
Late/Incremental Data Logic	Test logic for late-arriving or delta loads (SCD, watermark).
Partition Strategy	Confirm partition columns are chosen properly to avoid skew and optimize reads/writes.
🚀 4. Performance Optimization
Check	Description
Repartitioning Done Wisely	Avoid unnecessary shuffles; repartition based on data volume and downstream usage.
Broadcast Joins Used Correctly	Broadcast small lookup DataFrames instead of large shuffles.
Cache/Persist Used Judiciously	Used only when reused multiple times — unpersist when done.
File Sizes Tuned	Target output file sizes between 100–300 MB for optimal performance.
Avoid Collect/Show	Never use collect() or show() on large datasets in production code.
Adaptive Query Execution (AQE)	Enabled for dynamic optimization of joins and partitions.
🧰 5. Error Handling & Recovery
Check	Description
Try/Except Blocks	Wrapped critical logic in exception handling to log and fail gracefully.
Audit / Control Tables	Insert job metadata (start_time, end_time, record_count, status, error_msg).
Idempotent Design	Pipeline can be safely re-run without causing data duplication or corruption.
Checkpoint / Commit Strategy	For streaming or SCD jobs — confirm checkpoints, commit logic handled properly.
📦 6. Testing in Lower Environment
Check	Description
Unit Tests Passed	Test key transformation functions independently.
Integration Tests	Test full data flow with staging → silver → gold transitions.
Data Volume Simulation	Run with realistic data sizes to test performance and scalability.
Validation Reports	Validate record counts, totals, business rules between source and target.
🕒 7. Scheduling & Orchestration
Check	Description
Job Parameters in Airflow/ADF	Ensure DAG or ADF pipeline parameters are dynamic (dates, table names).
Dependency Handling	Upstream/downstream job triggers defined correctly.
Retry & Timeout Policies	Defined in Airflow DAG or Databricks job settings.
Alerts & Monitoring	Email/Slack alerts configured for failures and long runtimes.
🧑‍💻 8. Access Control & Security
Check	Description
Least Privilege Access	Ensure Spark service principals / ADF Managed Identity have minimal required access.
Data Encryption	Check encryption at rest and in transit (ADLS, DBFS).
Row/Column-Level Security (if applicable)	Enforced for sensitive datasets.
Audit Logging	Access logs and operation logs enabled for traceability.

Check	Description

When a PySpark Databricks pipeline fails during production deployment, debugging must be structured and fast — since production downtime directly impacts business SLAs.

Let’s go step-by-step with a practical and systematic debugging checklist you can follow 👇

🧠 1️⃣ Identify Where the Failure Occurred

Before diving into logs, first identify which layer or stage failed:

Layer	Typical Issue Examples
Deployment Level	Wrong config, missing dependency, cluster misconfiguration
Spark Job Execution	Logic error, schema mismatch, null pointer, data skew
Data Layer (Input/Output)	File missing, bad path, permission issue, empty dataset
Orchestration Layer (ADF/Airflow)	Connection failure, variable not passed, timeout, dependency misfire.

2️⃣ Check Databricks Job Run Logs

If your pipeline runs in Databricks, go to:

Databricks → Jobs → Runs → View Logs

Then inspect:

🔸 a) Driver Logs

Shows Python exceptions and Spark execution plan.

Common signs:

AnalysisException: table/path not found

Py4JJavaError: Spark or JVM-level error

MemoryError or ExecutorLostFailure: cluster or data volume issue

🔸 b) Cluster Event Logs

Check if the job failed before the Spark code started (e.g., environment setup, library install failure).

🔸 c) Spark UI (Job → Stages → Executors)

Identify whether failure occurred at read, transformation, or write stage.

Check shuffle, stage retries, skew, OOM, etc.

💡 Tip: Databricks automatically links to the Spark UI in the “Run Details” page — use that to pinpoint the exact stage.

⚙️ 3️⃣ Review Exception Trace in Logs

Typical PySpark failure categories:

Error Type	Root Cause	Fix
AnalysisException	Missing path/table/schema mismatch	Validate input paths & table schema
NullPointerException	Null data or missing columns	Handle nulls, validate schema
FileNotFoundException	Input file not found	Check ingestion or path variable
Job aborted due to stage failure	Data skew, OOM, shuffle spill	Tune partitioning, memory, caching
Timeout / Task not serializable	Wrong broadcast, circular reference	Optimize serialization or restructure code
Permission Denied	ADLS / Hive ACLs	Update storage ACLs or service principal role
ModuleNotFoundError	Missing Python dependency	Reinstall via cluster init script or %pip install
🧩 4️⃣ Check Configuration & Environment Differences

Common between lower → prod migrations:

Issue	What to Check
Wrong Config Path	Verify production configs (YAML/JSON/param tables) used correctly
Secret Scope / KeyVault Access	Check secret name differences between environments
Cluster Version	Spark runtime version mismatch (e.g., 12.2 → 13.3)
Library Version	PySpark, Pandas, JDBC, etc., version mismatch
Path Variables	Ensure /mnt/prod/... not pointing to /mnt/dev/...
Job Parameters	Missing or incorrect runtime arguments (date, table name, etc.)

⚠️ Tip: Always maintain separate config files per environment and verify environment variables before deployment.

🧱 5️⃣ Validate Data & Schema

Data-related issues are the #1 cause of production Spark job failures.

Steps:

Check source data presence

dbutils.fs.ls("/mnt/prod/staging/table_name/")


Validate schema consistency

df = spark.read.parquet("path")
df.printSchema()


Compare with expected schema (from lower env or metadata).

Check for bad records / corrupt files

spark.read.option("badRecordsPath", "/mnt/prod/bad_records").csv(path)

🧰 6️⃣ Check Airflow / ADF Logs (if orchestrated)

If Databricks job triggered via Airflow:

Check Airflow logs → task instance → logs tab
You’ll see Databricks API error (e.g., HTTP 403, Invalid Job ID, JobRunTimeout).

If via ADF:

Check Monitor → Pipeline Runs → Output → Error Message

Common failures:

Linked Service credential expired

Get Metadata / Lookup activity returned null

Databricks token expired

🧾 7️⃣ Enable & Review Audit Tables

If you maintain control/audit tables (highly recommended):

Column	Description
job_name	Pipeline name
batch_date	Execution date
start_ts / end_ts	Job timings
status	SUCCESS / FAILED
error_msg	Captured exception
record_count	Input/output counts

You can then easily filter failed jobs:

SELECT * FROM audit_table WHERE status = 'FAILED' ORDER BY end_ts DESC;

🔄 8️⃣ Run in Debug Mode or Replay in Lower Env

Once you identify the failing notebook/script:

Run the same code block in dev/UAT with prod parameters.

Use try/except in notebook to print variable values & stage counts.

Example:

try:
    df = spark.read.parquet(prod_path)
    print(f"Records read: {df.count()}")
except Exception as e:
    print(f"Error reading file: {str(e)}")


This helps confirm if issue is data-specific or environment-specific.

🧑‍💻 9️⃣ Cluster & Resource Issues

Check if cluster terminated mid-run (Databricks UI → cluster logs).

Check autoscaling, memory, and disk spill metrics.

Increase spark.driver.memory or spark.sql.shuffle.partitions if necessary.

📢 🔟 Communicate & Document

Once you isolate the cause:

Update your runbook or incident record with:

Root cause

Resolution steps

Preventive action

Add monitoring or alert for early detection next time.

🚦 Example Debug Flow
Deployment failed → Check Databricks run logs →
Driver logs show AnalysisException → Root cause = wrong ADLS path →
Fix config → re-run in UAT → verify audit table → redeploy.

Explain Star vs Snowflake Schema

Star schema consists of a central fact table connected to denormalized dimension tables, optimized for query performance and simplicity. Snowflake schema normalizes dimension tables into multiple related tables, reducing redundancy but increasing join complexity. In modern analytics systems, Star schema is typically preferred for reporting due to better performance and ease of use, while Snowflake is used when storage optimization and strict normalization are priorities.

Stateless Stream Processing

📌 In stateless processing, each micro-batch is processed independently without remembering any past data.

Key Points

No information stored from previous batches.

Output depends only on the current input.

Simpler & faster processing.

Less memory required.

Easy to scale horizontally.

Example Use Cases
Use Case	Example
Filtering	df.filter(col("amount") > 1000)
Mapping/Transform	df.select(...)
Basic Aggregation per batch	df.groupBy("city").count() (without watermark)
Example Code
stream_df = spark.readStream.format("kafka")...

filtered_df = stream_df.filter(col("status") == "SUCCESS")

query = filtered_df.writeStream.format("console").start()

2. Stateful Stream Processing

📌 In stateful processing, Spark maintains state across micro-batches — remembers previous results to compute the next.

Key Points

Maintains past information using state store.

Used for running aggregates & event-time processing.

Requires checkpointing.

Needs watermarking for cleanup.

Example Use Cases
Use Case	Example
Running total	Count of events per user over time
Sessionization	User session tracking by timeout
Deduplication	Remove duplicate events
Late data handling	Event-time windowed aggregation
Example Code: Windowed aggregation
aggDF = streamDF \
    .withWatermark("event_time", "10 minutes") \
    .groupBy(window(col("event_time"), "5 minutes"), col("user_id")) \
    .count()

query = aggDF.writeStream.outputMode("update") \
    .option("checkpointLocation", "path/to/chk") \
    .format("console").start()

🆚 Summary Table
Feature	Stateless	Stateful
Remembers past?	❌ No	✔ Yes
Uses checkpointing?	❌ No	✔ Required
Handles late data?	❌ No	✔ Yes (with watermark)
Performance	Fast	Slower
Aggregations	Simple per batch	Running / window / session
Memory usage	Low	High
Interview Short Answer 📝

Stateless processing handles each micro-batch independently without storing previous results. Examples are filtering, mapping, basic aggregation.
Stateful processing maintains state across batches and is used for running aggregations, window functions, sessionization and deduplication. It needs checkpoint and watermark to manage state.
